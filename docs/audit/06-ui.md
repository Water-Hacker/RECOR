## 8.1 Focus indicators on every interactive element

**Verified.** `applications/declarant-portal/src/styles/index.css` lines
22-25:

```css
*:focus-visible {
  outline: 3px solid var(--color-recor-accent);
  outline-offset: 2px;
}
```

The universal `*:focus-visible` rule paints a 3px gold (`--color-recor-accent`
= `#c8b067`) outline with 2px offset on every keyboard-focusable element.
Tailwind's `focus:ring-2` is layered on top of the wizard input class
(`field.tsx` line 14-15) and the locale selector (`App.tsx` line 92),
giving belt-and-braces visibility even in browsers that ignore the
universal rule.

Spot-checked interactive elements:
- Wizard step 1 inputs (`EntityStep.tsx`): inherit `inputCls` â†’
  `focus:border-recor-deep focus:ring-2 focus:ring-recor-accent`. âś“
- Wizard step 2 owners-row inputs (`OwnersStep.tsx`): same. âś“
- Add/Remove owner buttons (`OwnersStep.tsx` lines 57-69, 170-180):
  rely on universal `*:focus-visible`. âś“
- Wizard nav Back/Forward (`wizard/index.tsx` lines 416-444): same. âś“
- Sign+Submit button (`SignStep.tsx` line 75-88): same. âś“
- Locale `<select>` (`App.tsx` line 83): per-element Tailwind classes
  `focus:border-white focus:outline-none focus:ring-2 focus:ring-white/40`.
  **Caveat:** `focus:outline-none` removes the universal outline; the
  ring class restores a visible indicator but only for the white-on-deep-blue
  header context. Acceptable, but see 8.6.A.
- `DraftResumeBanner` Resume + Discard buttons: inherit universal. âś“

**Doctrine note (D12 production-grade):** the universal `*:focus-visible`
selector is the load-bearing primitive. Any future component that
explicitly sets `outline: none` without restoring an equivalent indicator
breaks the contract. Recommend an ESLint rule promoting
`jsx-a11y/no-static-element-interactions` to error and adding a
stylelint rule banning bare `outline: none`. Filed as follow-up.

## 8.5 `DraftResumeBanner` â€” modal focus trap

**The banner is intentionally NOT modal.** From source:

```tsx
<aside role="status" aria-live="polite" ...>
```

It uses `<aside>` + `role="status"` + `aria-live="polite"` so the
content is announced once, the declarant retains keyboard focus, and
the wizard remains usable without dismissing the banner. This is the
correct pattern for a non-blocking offer-to-resume â€” a focus trap
would be incorrect because the declarant can legitimately ignore the
offer and proceed to fill the form. The banner closes on Resume or
Discard.

This is consistent with the existing R-PORT-5 audit's classification
of A-2026Q2-3 (recommended `aria-describedby` linkage on the buttons)
as backlog rather than gating. âś“

**Caveat:** because the banner uses `aria-live="polite"`, a screen-reader
user hearing the announcement does NOT automatically gain focus on the
Resume button. A keyboard-only user must Tab to find it. Document and
accept; promoting to focus-trap would be incorrect.

## 8.6 Locale switching defects (NEW IN THIS PASS)

### 8.6.A `<html lang>` is hardcoded to `en`

**Severity:** High (WCAG 3.1.1 Level A â€” Page Language).

`applications/declarant-portal/index.html` line 2:
```html
<html lang="en">
```

The default platform locale is `fr` (French is the legal primary per
Architecture V4 P17 Â§ Declarant Portal i18n). The i18next configuration
in `src/i18n.ts` sets `fallbackLng: 'fr'` and the detector resolves
`localStorage['recor.locale']` first, `navigator.language` second. On
fresh load with a French browser, the page renders entirely in French
but the `<html lang>` attribute claims English. Screen readers will
choose the wrong pronunciation profile and apply English hyphenation
to French content.

**Compounded violation:** the locale switcher in `App.tsx` line 86-94
calls `changeLocale()` which mutates `i18next.language` and rerenders
the React tree. It does NOT touch `document.documentElement.lang`. A
declarant who switches from French â†’ Pidgin still has `<html lang="en">`.

**Confirmation via grep:**
```
$ grep -RIn "documentElement.lang" applications/declarant-portal/src
(no matches)
```

**Remediation (single-line fix in `i18n.ts`):**
```ts
i18n.on('languageChanged', (lng) => {
  document.documentElement.lang = lng;
});
```
Plus update `index.html` to use a runtime placeholder or default `fr`
to match the platform's legal default. Filed as follow-up; gating for
launch.

### 8.6.B `LocaleSelector` `<select>` removes universal focus outline

**Severity:** Medium (WCAG 2.4.7 Level AA â€” Focus Visible).

`App.tsx` line 92 sets `focus:outline-none focus:ring-2 focus:ring-white/40`.
The ring opacity is 40%; against the deep-blue header (`--color-recor-deep`
= `#082567`) this restores a visible indicator at the keyboard-focus
moment, but the 40% opacity reduces the contrast below the 3:1 ratio
the universal `outline: 3px solid var(--color-recor-accent)` would
provide. Not flagged by axe (axe only checks the color-contrast rule
on text + interactive elements, not on focus indicators specifically).

**Remediation:** Drop `focus:outline-none` and let the universal
`*:focus-visible` rule apply, OR raise the ring opacity to 100% and
use a contrasting hue. Filed as follow-up; non-gating.

### 8.6.C `aria-live` regions exist for status announcements

Cross-referenced and verified:
- `<aside role="status" aria-live="polite">` on the drafts-unavailable
  notice and the resume banner.
- `<div role="status" aria-live="polite">` on `VerificationStatus`.
- `<span role="alert">` on every per-field validation message.

The mixed use of `role="status"` (live region with polite priority) +
`aria-live="polite"` is intentional â€” `role="status"` implies
`aria-live="polite"` but the explicit attribute is the belt-and-braces
for AT engines that historically were inconsistent on implicit live
priorities. âś“

## 8.10 Reduced-motion

**Not documented and partially observable.**

`grep -RIn "reduced-motion\|prefers-reduced-motion\|motion-safe\|motion-reduce"
applications/declarant-portal/` returns zero matches. The CSS file
contains no `@media (prefers-reduced-motion)` rule.

**However**, the only motion in the portal is the Tailwind `transition`
class on Back, Forward, and Sign+Submit buttons (`SignStep.tsx`
line 81, `wizard/index.tsx` lines 422, 436). Tailwind's `transition`
shorthand expands to `transition-property: color, background-color,
border-color, ...` with a 150ms duration â€” well below the WCAG
"three flashes per second" threshold, and not vestibular-triggering.
There are no slides, no auto-playing carousels, no parallax.

**Recommendation:** add an explicit `@media (prefers-reduced-motion:
reduce) { *, *::before, *::after { transition-duration: 0.01ms !important;
animation-duration: 0.01ms !important; } }` rule to `styles/index.css`
as a defensive measure for future motion additions. Documented as
non-gating; the absence of motion makes it a forward-looking hygiene
fix, not a current WCAG violation.
